# Future TODO 参考详解

这份文档只列**还没做**的方向。已经落地的内容直接看代码（`team-role-*.js`、`render-*.js`、`analysis.js` 等）即可，不在这里重复维护。原 `docs/optimization-suggestions.md` 的待办条目已合并进来。

新增 TODO 时遵循两条规则：

- 用户已经验证过、暂时不做的方向写明「为什么暂不做」。
- 不写"已完成"段落——做完了就把对应条目从这里删掉。

## 一、产品方向（参考站对齐）

### 1. 桌面端操作台分区

当前项目有主工作区和分析子页签，但还没到参考站「编辑台」深度。

- 左侧固定为当前选中队员的预览卡，而不是只固定整队列表。
- 右侧编辑区拆成更明确的工作流标签：`Moves / Speed / EVs / Damage / Analysis / Roles / Cores`。
- 桌面端和移动端使用不同的信息密度：桌面端强调并排编辑，移动端保留分段式堆叠。
- 分页切换不只是内容显隐，还要让用户明确知道"当前正在编辑哪一只"。

为什么暂不做：现有数据模型以"导入 Showdown 配置后整体浏览/选队"为中心，不是单只精灵编辑器。直接照搬会把"配置库驱动"的产品方向改成"逐只手工组队器"。

### 2. Coverage 的下一步

- 在 Coverage 面板内直接显示"Suggested Covers"：从当前配置库里给出 3 个候选，并支持轮换。
- 把"当前队伍已经能处理哪些弱点"与"仍未覆盖哪些弱点"分成两块。
- 推荐时增加"少重复属性优先"的显式开关，而不是只体现在分数里。

### 3. Cores 的下一步

- 增加"三人 core"模式，而不是只看双人。
- 将 core 评分拆分成多项：`补主要弱点 / 免疫补位 / 共享弱点 / 单属性覆盖 / 双属性覆盖 / 职能互补`。
- 把"真实候选"继续深化到招式级协同，而不只停留在属性 / 职能 / 速度模式层。
- 核心对建议没有"替代对"的可视化比较。

### 4. Speed 工作台

- 单只精灵的速度配置器：`EV / IV / Nature / Tailwind / Web`。
- 当前选中精灵在 tier 中的实时定位。
- "我的速度"和"常见满速线"同表显示。
- 自动滚动到自己的位置。

实施约束：保持 Champions 规则；只引入真正需要的输入项；和当前配置编辑器共用数据模型。

### 5. EV / IV 实时编辑器

- 当前配置的六维点数可视化。
- Champions 点数上限提示。
- Nature 对最终面板的影响。
- 最终能力值实时刷新。

### 6. Damage 计算器后续

继续可做：

- 更强的批量矩阵筛选与导出。
- 更完整的专题面板（速度 / 点数 / 伤害联动）。
- 更少 shim、更多纯函数化的伤害核心封装（清理 `damage-core-worker.js` 的 jQuery shim 造成的隐式行为）。
- 太晶 / 动态 / Gigantamax / Terapagos Stellar 字段在计算参数里有，但 UI 没暴露。

### 7. 随机组队生成器

不是从配置库随机抽 6 只，而是「带约束的推荐构队」：优先补承伤缺口 / 优先补缺失职能 / 避免过度重复属性 / 考虑速度模式。

进一步：风格预设（进攻 / 平衡 / 偏受）/ 锁定已有成员后重抽 / 指定起手核心。

### 8. 导入导出与保存体系

- 更明确的导入结果摘要：成功几条、跳过几条、失败几条。
- 保存快照的时间戳和版本号。
- 从保存队伍反向恢复筛选条件或视图状态。
- 「队伍备份到 JSON 文件」入口（部分完成，仍可继续）。
- 通过 URL 分享队伍。
- 对战结果记录。
- 配置合法性校验（部分完成）。
- 默认预设的"分享日期"在 UI 完全隐形（`paste_sets_champions_ma.json` 的 `selected.source.dateShared` 已经写入，但 UI 还没消费）。
- 导出 / 导入弹窗仍是 `<details>` + `<textarea>`，缺乏分屏对比。

## 二、职能识别仍剩的尾巴

- **太晶职能**：M-A 赛季禁太晶，整组规则（太晶爆破手 / 太晶防守转型 / 太晶反粉末 / 太晶反击掌 / 太晶抗性修正）等下个允许太晶的 Champions 赛季再实现。底层 `getEffectiveTypes` 已经支持太晶语义，可以无缝接入。
- **counter 判定的 effective speed 升级**：当前 `team-role-counter.js#effectiveSpeed` 只算 base + 20 + points + nature + Choice Scarf；没有处理 Tailwind / Trick Room 场地、Protosynthesis / Quark Drive / Swift Swim / Chlorophyll 等条件特性的速度倍率。下个迭代可以接 `battle-semantics.getEffectiveSpeed` 做完整版。
- **环境对位的 effective-speed 模拟器**：`corecheck` 现在用 2HKO + 抗一击近似 winrate，可以升级为 16 回合 / 4 回合的迷你模拟，但只在用户报「corecheck 误判」时再做。
- **特性评分表**：`ability-scores.js` 的评分仍粗糙、且历史上有事实错误，需要按 M-A 环境逐项复核。

## 三、算法语义补强（来自原优化建议）

### 3.1 双打机制建模（部分完成）

- 对带特殊类型作用的招式（Freeze-Dry / Flying Press 类反逻辑命中）的例外仍未完整建模。
- 威吓 / 顺风 / 麻痹 / Trick Room 等状态对速度的影响——速度排序模块支持，但是评分/推荐链路里仍是粗略近似。
- 先制招式 / 优先级建模仍部分缺失（`team-role-counter.js` 已经接进 priority，但攻击覆盖 / counter 选择里还没全面用）。
- Wide Guard / Follow Me / Helping Hand / 保护预测在打分里几乎缺席（部分接进职能 tag，但推荐打分还没用）。

### 3.2 推荐打分

- **推荐权重使用线性加法**，不能表达"必须都满足"——目前是分项相加。下一步可以加阈值否决（某项过低直接出局）。
- **`qualityBreakdown` 分项单位可疑**——分项之间量纲不一致，需要 normalize。
- **先发对推荐没有使用搜索剪枝，且从不缓存**（部分完成）——大队伍下计算量爆炸。
- **先发推荐只给分数和"目标"两列，不给动作剧本**——可以加"第一回合应该做什么"的简短指引。

### 3.3 ruleset 抽象

- `recommendations.js` 的 Mega 限制在 M-A 环境下是正确行为，但 ruleset（哪些环境允许 Mega / 太晶 / Z 招）应该拆成可切换的配置，不是写死。

### 3.4 解析器与数据

- **`showdown.js` 解析器不够健壮**（部分完成）——边角格式（缺 nature 行、tab vs space、特殊角色）仍会失败。
- **使用率（Usage）数据只用单一档次**——目前固定 1500，不同 rating 视图没有切换。
- **数据 JSON 没有版本标识**——无法在客户端做版本兼容判断。
- **数据合并逻辑与中日译名解耦不够**——`data.js` 里把翻译和数据合并耦合在一起。

### 3.5 概念混淆

- **"队伍"和"单配置"概念混淆**——配置库（每条单独一只）和队伍（六只组合）在导入导出 / 持久化 / UI 命名上时常互相串。

## 四、UI / 可视化

- **核心对建议**没有"替代对"的可视化比较。
- **分析维度切换**（Coverage / Roles / Cores）不保留上下文（部分完成，五期已增强但仍可继续做）。
- **图标方案切换**后一些隐式状态不联动。
- **移动端布局**：六只队伍卡片在窄屏挤压（部分完成）。
- **配置库的"差异预览"在卡片上没暴露**——和 `library-compare` 视图的差异显示没接进卡片层。
- **配置卡的 hover 信息密度浪费**——hover tooltip 还是一层信息，可以利用空间显示更多。
- **主视图切换的"在做什么"指示缺失**（部分完成）——View Transition 已接，但当前阶段标识还不明显。

## 五、可访问性 / i18n / 响应式

- **Modal 没有 focus trap**——背景仍可 Tab。
- **颜色语义**缺少无障碍备选（红绿色盲不友好）。
- **`title` 工具提示在移动端不可用**——长按、`focus-visible` 等替代路径仍未铺开。
- **切换语言后动态渲染的内容要完整重渲染才生效**——i18n 应用仍局部失败。
- **没有复数形式与数字格式**——`{count}` 直接拼接没经过 ICU。

## 六、性能 / 渲染 / 打包

- **启动时并行拉 9 个 JSON**，无 preload 与分级加载——首屏阻塞较重。
- **`renderSpeedTiers` 对每条变体重新调 `spriteMarkup`**——重复计算可以做成 memoize。
- **`buildSpeciesTemplateConfigs`** 对所有可用 species 生成 3-4 个模板——首次构建慢。
- **没有 bundler**——import 图强依赖 HTTP 请求数，在弱网下可见加载抖动。
- **大量 ESM 里的辅助函数在渲染层重复实现**——可以抽公共 util。
- **damage 计算 worker 的 `importScripts` 阻塞加载**——首次伤害扫描会卡。

## 七、错误处理 / 持久化

- **错误处理风格不一致**——有的 throw、有的 console.warn 静默吞、有的 toast，路径不统一。
- **`localStorage` 写入失败时没有降级**——quota 用尽会静默丢数据。
- **没有"队伍备份到 JSON 文件"能力**（部分完成）——只能手抄 Showdown 文本。

## 八、不建议直接照搬的内容

- 整套深色科幻视觉包装、视频背景、随机队生成器的完整 UI、与当前数据模型不兼容的逐只精灵编辑器、大量面向 Smogon / tier 游戏模式的规则判断。
- 当前项目的定位是"静态配置库驱动的 Champions 队伍分析器"。直接照搬会让产品方向偏掉，也会把代码复杂度拉高很多。
